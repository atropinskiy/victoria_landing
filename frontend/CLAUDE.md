# Инструкции для проекта

## Стек

- Next.js (App Router), TypeScript, Tailwind CSS
- shadcn/ui + Radix для компонентов
- next-intl для локализации
- class-variance-authority (cva) для вариантов компонентов

## Общие правила

- Строгая типизация: никаких `any`, если можно вывести тип — выводи
- Компоненты — функциональные, с именованным экспортом (`export function X()`), не default export
- Стилизация только через Tailwind-классы, никаких инлайн-style и отдельных .css файлов без необходимости
- Иконки — через компоненты из `@/components/icons`, не хардкодить SVG в JSX
- Все интерактивные элементы — с корректными aria-атрибутами (aria-label, aria-expanded и т.д.)

## Структура и конвенции

- Хуки — в начале компонента, до early return
- Пропсы — деструктурировать в сигнатуре функции
- Условный рендеринг — через && или тернарник, не через переменные-флаги без необходимости
- Файлы компонентов — kebab-case, сами компоненты — PascalCase

## Перед тем как писать код

- Если в проекте уже есть похожий компонент/паттерн — следуй ему, не изобретай новый подход
- Проверяй существующие utility-классы и переменные темы (cn(), buttonVariants и т.п.) перед тем как писать кастомный CSS
- Если что-то неочевидно из контекста — сначала посмотри соседние файлы, а не спрашивай

## Что НЕ делать

- Не добавлять новые зависимости без явного запроса
- Не переписывать чужой код "для красоты", если не просили — только то, что относится к задаче
- Не оставлять закомментированный мёртвый код и console.log
- Не использовать !important без крайней необходимости

## Тесты и проверка

- После изменений запускать `npm run lint` и `npm run typecheck` (или соответствующие команды проекта)
- Если добавляешь новый компонент — проверяй, что он работает в mobile и desktop вариантах (проект адаптивный)
